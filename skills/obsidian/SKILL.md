# Obsidian (second brain)

The vault directory is named `ossians-second-brain-sync`. It usually lives at `~/ossians-second-brain-sync`, but sometimes under `~/Developer/`. **Resolve it first** rather than assuming a path:

```bash
VAULT=$(for d in ~/ossians-second-brain-sync ~/Developer/ossians-second-brain-sync; do [ -d "$d/.obsidian" ] && echo "$d" && break; done)
```

It is plain markdown — **work on the files directly** with Read / Grep / Glob / Edit. Do not use the `obsidian` CLI or require the app to be open. The only thing files can't do is *render* a `.base` or `dataview` query (see [Escape hatch](#escape-hatch)).

## First: read the vault's own rules

The vault documents its own conventions and they are the source of truth — not this skill. Read them before writing anything:

- `$VAULT/AGENTS.md` — structure, PARA + Zettelkasten model, maturity system, Readwise workflow, naming + frontmatter rules.
- `$VAULT/LINKING.md` — wikilink and tagging conventions.

This skill is just a fast orientation + the guardrails that are easy to get wrong.

## Layout (PARA outside, Zettelkasten inside Resources)

- `1. Projects/`, `2. Areas/` — actionable. Frontmatter `type: project|area`, not a maturity tag.
- `3. Resources/` — the knowledge layer:
  - `Notes/` — atomic notes (default home for new notes), one idea per file.
  - `daily-notes/` — `YYYY-MM-DD.md`, **no frontmatter**.
  - `Readwise/` — generated by the Readwise plugin; **read-only, never edit bodies**.
  - `attachments/` — images, embedded via `![[file.ext]]`.
  - `*.base` — Obsidian Bases views (see [base_syntax.md](references/base_syntax.md)).
- `4. Archive/` — inactive.
- `moc/` — Maps of Content (frontmatter tag `moc`).
- `templates/` — note templates (Templater `<% %>` syntax).

## Core rules (easy to get wrong)

- **Links:** exact, case-sensitive `[[Note Title]]`. No aliases (`[[Note|Display]]`), no section links (`[[Note#Heading]]`).
- **Tags:** frontmatter only — no inline `#hashtags`. A note carries exactly one maturity tag: `seedling | budding | evergreen`. MOCs also add `moc`.
- **New notes** default to `3. Resources/Notes/`. Title Case filename matching the `# H1`; frontmatter `created: YYYY-MM-DD` and `tags: [seedling]`.
- **Daily notes:** no frontmatter; entries start with `HH:MM`, nested bullets, link liberally.
- **Templater won't run** on direct file writes — copy the matching `templates/*.md` frontmatter and fill placeholders yourself (e.g. `created` = today's date).
- **Never touch `.obsidian/`.** If you ever genuinely must edit `.obsidian/plugins/*/data.json`, Obsidian must be fully quit first (⌘Q) — it overwrites those files on exit and silently reverts live edits.
- **Readwise:** new insights become atomic claim notes in `3. Resources/Notes/` (see the `readwise-claim` template), never edits to `Readwise/` bodies.

## Common operations

```bash
# $VAULT resolved as above
# Notes linking to a topic
grep -rl "\[\[Topic Name\]\]" "$VAULT/3. Resources/Notes/"
# Notes with a maturity tag (frontmatter)
grep -rl "^  - evergreen$" "$VAULT/3. Resources/Notes/"
# Today's daily note
echo "$VAULT/3. Resources/daily-notes/$(date +%F).md"
```

Use **Glob** to find notes by name, **Grep** for content/links/backlinks, **Read** to load, **Edit/Write** to change. Backlinks = grep for `[[This Note]]` across `3. Resources/Notes/` and `moc/`.

## Escape hatch

Files give you note *content* but not the *rendered result* of a `.base` view or a ` ```dataview ` block — those are computed by plugins at runtime. If the user needs the actual rows such a query returns, reconstruct the query logic yourself from the `.base` / code-block definition (it's just filters + sort over frontmatter — see [base_syntax.md](references/base_syntax.md)). Only if that's insufficient and Obsidian is open, fall back to the `obsidian` CLI for that one read. Don't reach for the CLI otherwise.
