# App Store Screenshots

Research by [DesignerAnts](https://www.designerants.com) on screenshot copy that converts.

## The 70/30 Rule

**70% of screenshot effectiveness comes from the text overlay** — not the UI. One app saw an **80% conversion lift** just from rewriting the copy on existing screenshots.

Your UI is evidence. Your text is the argument.

## The Wrong Question

Feature descriptions answer the wrong question. A user scanning your App Store page isn't asking "what does this app do?" They're asking **"is this for me?"** Feature lists don't answer that.

## Specificity Sells

The reason this works: specificity makes the promise real.

- "Productivity app" is invisible.
- "Never lose a meeting note again" is a reason to download.

The more precisely you describe the user's life *after* your app, the more they can see themselves using it.

## The Hand Test

Cover your UI with your hand and read only the text. Does it tell a story? Or does it list features?

## The Rewrite Mindset

**Stop describing features. Show what changes for the user.**

| Instead of... | Write... |
|--------------|----------|
| "Customizable dashboard" | "See everything that matters, at a glance." |
| "Workout tracking" | "You'll never forget what you lifted again." |
| "Notes → Slides. Instantly." | "Stop designing slides. Write notes. Start presenting." |
| "Full screen always readable" | "Stay focused while you present. Clean slides. No distractions." |
| "Slides over any app" | "Stay in your flow. Present without breaking your momentum." |
| "Auto-numbering that works" | "Never fix slides manually again. Your structure stays clean automatically." |

## The Screenshot Sequence

Your screenshots should only make sense in order. If they work in any sequence, you have a catalog, not a story.

### Screenshot 1 — Name the Pain
Their frustration, **before** they found you.
> "Buried in notes you'll never find again?"

### Screenshot 2 — State the Shift
What changes **when they use** your app.
> "Everything you capture, organized automatically."

### Screenshot 3 — Show Proof
Numbers, users, concrete result.
> "Used by 10,000 developers every day."

### Screenshots 4–5 — Feature Delivery
The one or two capabilities that actually deliver the promise from Screenshot 2.

**Each screenshot does one job. One message.** If it needs two sentences to explain, split it into two screens.

## The 8-Word Rule

Write the headline for each screenshot **before** designing the screen. If you can't say the change in 8 words, you don't understand it well enough yet. Then let the UI behind it serve as the visual proof.

**Treat the words as the product. Everything else is supporting material.**

## Before/After Case Study: Super Easy Slides

**The launch copy (mistakes):**
- Notes → Slides. Instantly.
- Full Screen Always Readable.
- Slides Over Any App.
- Auto-Numbering that Works.

**Problems with this:**
- Reads like feature labels, not headlines
- No clear user pain or motivation
- Nothing grabs attention or stops the scroll
- Assumes the user already understands why this matters
- Described the product—but didn't sell it

**The rewrite (outcomes):**
- Stop Designing Slides. Write notes. Start presenting.
- Stay Focused While You Present. Clean slides. No distractions.
- Stay In Your Flow. Present without breaking your momentum.
- Never Fix Slides Manually Again. Your structure stays clean automatically.

**Why the new version works:**
- Leads with outcome, not implementation
- Makes sense instantly (even to non-users)
- Connects to real pain: time, focus, clarity
- Uses the feature as support—not the headline

## Quick Checklist

- [ ] Headline works with UI covered (hand test)
- [ ] Each screenshot = one message
- [ ] Sequence tells a story (pain → shift → proof → features)
- [ ] Headlines are 8 words or fewer
- [ ] Features serve outcomes, not the other way around
- [ ] Text sells, UI proves
