# Slot Strategy — What Goes In Each Screenshot Slot

After benefit discovery, use this to decide which benefit lands in which slot. The order matters more than the individual screenshots: users spend **3–6 seconds** on a product page before deciding, and the **first 3 screenshots** (visible without scrolling on the search result and product page) drive ~80% of the conversion decision.

What users are asking, in order:

1. "Does this solve my problem?" → Slot 1
2. "Is it easy to use?" → Slots 2–3 (UI clarity)
3. "Is it worth downloading?" → Slots 4–10 (depth, trust, differentiation)

## 10-Slot Framework

### Slot 1 — The Hook

The single most important screenshot. Must answer "what does this app do and why should I care?" in one glance. Pick the hook pattern that fits the app:

| Pattern | Use when | Example |
|---------|----------|---------|
| **Benefit headline + key UI** | Most apps | "Sleep Better Tonight" + sleep tracking screen |
| **Before / After** | Transformation apps (fitness, finance, learning) | Cluttered inbox → zero inbox |
| **Social proof + UI** | Apps with real volume | "5M+ users trust us" + main screen |
| **Problem statement** | Pain-point-driven categories | "Tired of forgetting?" + reminder UI |

**Avoid in Slot 1**: welcome screens, login/signup, settings, generic UI without a hook.

### Slots 2–3 — Core Value

The 2 most compelling features after the hook, each with a benefit-driven caption. These complete the "above the fold" trio that decides most conversions.

### Slots 4–7 — Feature Showcase

One feature per slot, formatted as:

```
[Benefit Headline] + [Feature UI] + [Supporting detail]
```

Pick features that show depth, not duplicate the hook. If two benefits make the same promise, only one belongs here.

### Slots 8–9 — Trust & Differentiation

- Awards, press, ratings
- Comparison vs. category alternatives (clearly framed)
- Premium or unique-to-this-app features

### Slot 10 — Call to Action

Final reinforcement. Options:

- "Start your free trial"
- "Join [X] million users"
- Recap of top 3 benefits in one frame

## Text Overlay Rules

| Do | Don't |
|----|-------|
| Benefit-driven headlines ("Sleep Better Tonight") | Feature names ("Push Notifications") |
| 4–6 words per headline | Long paragraphs |
| Min 60px font, readable at thumbnail size | Small text that disappears at search-result scale |
| High contrast (text vs. background) | Text over busy UI |
| Consistent font + style across all 10 | Mixed fonts and sizes |

## Layout Patterns

**Portrait (default for most apps):**

- Device frame centered, text above or below
- Full-bleed UI with text overlay
- Split layout: text left, device right

**Landscape (games, video editors, immersive content):**

- Full-screen gameplay or content
- Minimal text overlay
- Action-focused moments

## Slot Plan Output Format

When committing to the plan before generation, write it out as:

```
Slot 1 — [Hook headline]
  Caption: "[benefit-driven text]"
  Screen:  [which app screen to feature]
  Layout:  [portrait centered / full-bleed / split]
  Pattern: [benefit + UI / before-after / social proof / problem statement]

Slot 2 — [Headline]
  Caption: ...
  Screen:  ...
  Layout:  ...

[repeat for all 10]
```

This is the bridge between Benefit Discovery and Generation in the main SKILL.md — pair each confirmed benefit with a slot, hook pattern, and screen before sending anything to Nano Banana.

## Design Brief (for handoff to a designer)

If the user is briefing a human designer instead of generating with Nano Banana, deliver:

- Color palette (from app brand)
- Font recommendations (with min size)
- Layout template (which of the 3 portrait patterns)
- Text overlay copy for all 10 slots (from the plan above)
- Device frame preference (modern frame / frameless / custom)
- Background style (solid brand color, gradient, photographic)
